> this file is the *how*, not a re-argument of the vision.

## Spec

Run workflows on push, in a limited-but-real form: on a successful ref advance,
execute the repo's workflow in an isolated container, keep logs, report pass/fail,
surface status in the UI. The strategic payoff (per the evolution note) is
**inverting the source of truth** — S3 + gitmote become the origin, GitHub a
mirror — and closing the loop so a push can **redeploy gitmote itself**, made
safe by the leased writer ([safety.md §1](../docs/architecture/safety.md),
[ops.md](../docs/ops.md)).

**Non-goals (MVP):** full GitHub Actions parity — marketplace, matrix builds,
service containers, remote caching, OIDC. ~80% of personal CI is "on push, run
steps in a container, pass/fail, keep logs"; build that, grow compat only where
the lack is felt.

## Current — the seams this reuses

CI does not bolt on; it hangs off primitives already built (the evolution note's
table, made concrete):

| CI need | Existing seam | File |
|---|---|---|
| Trigger (`on: push`) | The ref **CAS commit** — the one durable "a ref advanced" moment | [`Writer.handle`](../internal/githttp/receive.go#L189) — immediately after `CASRefs` succeeds (`receive.go:216`), before `Response{OK:true}`. There is already a `beforeCAS` hook; this adds an **after-commit, fire-and-forget** dispatch. |
| Read the workflow file at the pushed SHA | The browse reader from tasks 14/15 | [`repo.Tree` / `repo.Blob`](../internal/repo/browse.go) over a `Materialize`d dir — no new object plumbing. |
| Run / job state | An s3lite table, single-writer | [`internal/meta`](../internal/meta) — new `ci_runs` / `ci_jobs`, written **only by the leader** (like refs). |
| Logs & artifacts | Append-only blobs in S3, content-before-pointer | [`store.Store`](../internal/store) under a new `ci/` key space. |
| Runner → server report | An authenticated internal API + shared secret | Same discipline as the [hookrpc](../internal/hookrpc) push channel; runners **never touch s3lite directly**. |
| Status UI + log view | Admin-gated server-rendered pages + sanitized rendering | [`internal/webui`](../internal/webui) + [`internal/render`](../internal/render). |

**Runner substrate — Scaleway Serverless Jobs.** A finite task that scales to
zero, with a real Docker daemon + CPU the always-warm Container can't offer. The
**exact invocation pattern is proven in a sibling project** and directly
stealable — `~/dev/pan0.com/lib/scaleway/jobs.go`:

- `POST https://api.scaleway.com/serverless-jobs/v1alpha2/regions/{region}/job-definitions/{id}/start`
- Header `X-Auth-Token: <SCW_SECRET_KEY>`; body `{"environment_variables": { … }}`.
- The job **definition** is created once, out of band (`scw jobs definition create … cpu-limit … memory-limit … local-storage-capacity …`); the app injects **per-run env at start** (so one definition serves all repos/branches).
- **Fire-and-forget + webhook callback**, not polling: worker reads `JOB_ID`/`APP_URL`/`WORKER_SECRET` from env, `GET`s the job spec to claim it, runs, then `POST`s completion — all authenticated with a constant-time `WORKER_SECRET` compare. pan0's `handle_internal_jobs.go` (claim + idempotent `ApplyCompletion`) is the template for our internal API.
- No-op when the job-definition ID env is unset → local dev / tests run the whole flow with a stub, exactly as pan0 does.

## Stage 0 — Decisions to lock first (no code)

These gate the design; resolve before Stage 1.

1. **Engine — ✅ LOCKED: `act`** (not Woodpecker). Decisive reason is the mirror:
   `act` runs `.github/workflows` YAML (~80% Actions-compatible), so **one** CI
   definition runs both self-hosted *and* on the GitHub mirror — the break-glass
   redeploy uses the same workflow. Woodpecker would mean **two** definitions that
   drift, weakening "GitHub as a clean redeploy." Actions-compat is load-bearing
   here, not a preference. The runner image (Stage 4) is built around `act`'s
   engine (cf. Forgejo's `act_runner`).
2. **Workflow file location & format — ✅ settled by (1):** `.github/workflows/*.yml`,
   GitHub Actions YAML. No native format.
3. **Secrets model — ✅ LOCKED.** Per-repo CI secrets, encrypted at rest,
   injected into the runner env only at trigger time. Prior art:
   `~/dev/atmin.net` has mature crypto worth stealing *selectively* — but its
   **trust model is inverted from ours and that difference drives the design.**

   - **Trust model (the honest part first).** atmin.net is E2E: the server never
     sees plaintext, the master key is *derived from a user password client-side*.
     gitmote is the opposite — the **server must decrypt** to inject secrets into
     the runner, so it **holds the master key** (a container secret env var,
     alongside `GITMOTE_COOKIE_KEY` / AWS keys — see [ops.md](../docs/ops.md)).
     Therefore encryption-at-rest here buys exactly one thing: **a compromise of
     the S3 replica / DB snapshot (the litestream WAL, the object bucket) does not
     leak secrets, because the master key lives only in the container env, not in
     the replicated DB.** It does **not** protect against a compromised *running*
     server. Say this plainly in the safety doc — don't oversell it. No Argon2id,
     no password-KDF: there's no user password in this path.
   - **Primitive — port directly.** AES-256-GCM, **12-byte random IV per
     encryption** from `crypto/rand`, never reused (atmin.net's `backupEncrypt`,
     `web/src/lib/crypto.ts`). In Go: `crypto/aes` + `crypto/cipher` GCM.
   - **Per-repo subkeys — steal the HKDF-per-purpose pattern.** Derive a per-repo
     key from the master with HKDF-SHA256 (`info = "ci-secret:" + repoID`), so a
     single container-held master gives per-repo key isolation for free (mirrors
     atmin.net's `hkdfDerive(secret, info)`).
   - **Versioned envelope + O(1) rotation — steal the key-chain idea.** Store
     `{v, iv, ciphertext}` with a leading key version; rotate by **wrapping the old
     key under the new one** (a `key_chain`) rather than re-encrypting every secret
     — atmin.net's ADR-0012 / `key-chain.ts`. Lazy migration: rotation is O(1),
     old-secret reads walk the chain.
   - **One improvement over the prior art:** atmin.net's envelope does **not**
     bind `v`/name into the GCM tag (an acknowledged AAD gap). We should pass
     **AAD = (repoID, secretName, version)** to GCM, so a stolen ciphertext can't
     be replayed under a different repo/name/version.
   - **Storage:** a `ci_secrets` table in `meta` (s3lite) — single-writer,
     litestream-replicated like everything else; the master key is the one thing
     *not* in there. Never logged, never rendered (Stage 5 test asserts this).
4. **Runner clone auth — ✅ LOCKED.** A **per-run, read-only,
   repo-scoped, short-lived token**, minted by the leader at dispatch and revoked
   on completion. The runner clones over normal git-HTTPS
   (`https://x-access-token:<tok>@gitmote…/owner/repo`, checkout the SHA) — the
   **same authenticated path as any client**, no CI backdoor in the git handler.
   Rejected: a `WORKER_SECRET`-derived clone credential (shared across all runs →
   a leak grants broad standing access; violates least-privilege). Rejected:
   presigned-S3 / git-bundle hand-off (duplicates the git path, and the runner
   would need broad bucket creds).
   - **Cost:** a contained extension to the token model — the
     [tokens table](../internal/meta) gains nullable `expires_at`, a `repo_scope`
     (single repo), and a `read_only` flag; [`auth.Guard`](../internal/auth)
     honors them (expired / over-scoped → denied). Expiring, scoped tokens are
     generally useful beyond CI. TTL = run max-duration + margin; sweep expired
     tokens on the log-GC cadence. Lands as a small precursor sub-task to Stage 4.
5. **Log storage & retention — ✅ LOCKED.**
   - **Layout:** a new top-level `ci/` key space in the object store (peer of
     `objects/` and `meta/`): `ci/{repoID}/{runID}/{jobID}.log`. Artifacts, when
     they exist, go under `ci/{repoID}/{runID}/artifacts/` (out of MVP scope).
   - **Write:** MVP uploads the **complete log once on completion** (one PUT, like
     pan0's worker); the pointer is recorded in `meta` **after** the blob lands
     (content-before-pointer, unchanged). Live tailing / chunked append is a later
     want, not MVP.
   - **Caps (no silent truncation):** a per-log size cap (e.g. 10 MiB) so a
     runaway/malicious workflow can't fill S3; on hitting it, truncate with an
     explicit marker (the "more" discipline from task 14), never silently.
   - **Retention:** time-based default (**30 days**) plus keep-last-N per repo,
     swept by the **leader** — deletes old run rows and their `ci/` blobs together.
     Logs aren't sacred (a re-run regenerates them), so they need no
     litestream-grade durability; being in the same durable bucket is enough.
6. **Concurrency / cancellation — ✅ LOCKED.** MVP does **not** cancel
   in-flight runs. Dispatch is serialized by the single writer (runs enqueue in
   push order); execution is parallel on Scaleway. When a newer push supersedes an
   older run's ref, the older run is **marked `superseded`** in the UI but **runs
   to completion** — correctness comes from Stage 7's **latest-wins deploy guard**
   (a stale run deploys nothing), not from racing to cancel. Actively stopping a
   superseded Scaleway job (to save compute) and a per-repo concurrent-run cap are
   **follow-ups**, added only if compute cost or a push-storm is felt; rely on
   Scaleway's own job-concurrency limits until then.

## Staged plan

Each stage lands non-breaking and tested (golden **and** failure path, per
[CONTRIBUTING](../CONTRIBUTING.md)). Ordered so value appears early and the risky
surfaces (untrusted execution, secrets) come after the data model is solid.

### Stage 1 — Data model + dispatch seam (no execution)
- `ci_runs` / `ci_jobs` tables in `meta`; a `Runs` API mirroring `refs` discipline.
- Add an **after-commit** callback to `Writer` (sibling to `beforeCAS`), invoked
  after `CASRefs` succeeds. It records a queued run row and returns immediately —
  **never blocks or fails the push** (a failed dispatch is a *missed deploy, not a
  failed push* — the content-before-pointer rule, applied to CI).
- **Verify:** a ref-advancing push creates exactly one queued run; a rejected /
  non-fast-forward push creates none; a forced dispatch error leaves the push
  green.

### Stage 2 — Config discovery
- On dispatch, read the workflow file from the pushed commit's tree via the
  browse reader at the SHA. No file → no run. Malformed config → a run recorded as
  **failed with a clear message**, never a crash.
- **Verify:** repo with / without the file; a malformed file surfaces as a failed
  run, not a 500 or a panic in the writer.

### Stage 3 — Scaleway Jobs client + trigger
- New `internal/scaleway` `JobsClient.Trigger` — port of pan0's `jobs.go`
  (raw `net/http`, `X-Auth-Token`, `…/start`, env-var body). No-op when the
  job-definition ID is unset.
- Wire Stage 1's dispatch to call `Trigger` with `RUN_ID`, `APP_URL`,
  `WORKER_SECRET`, repo, SHA. Fire-and-forget.
- **Verify:** an httptest server asserts URL / `X-Auth-Token` / body shape; unset
  definition ID → no-op; a trigger error marks the run failed but the push stays
  green.

### Stage 4 — Runner image + internal report API
- `cmd/gitmote-runner` + `Dockerfile.runner`: claim the job
  (`GET /internal/ci/jobs/{id}`, `X-Worker-Secret`), clone at the SHA, run the
  engine over the workflow, stream logs to S3, `POST …/complete`.
- `internal/webui` (or a dedicated internal handler): claim + **idempotent**
  completion (pan0's `ApplyCompletion` shape), constant-time secret compare,
  runner **never** touches s3lite.
- **Verify:** end-to-end against a test server with a local/fake engine;
  completion is idempotent (double-POST is a no-op); missing/incorrect secret →
  401; a run that never reports is reconcilable (timeout → failed).

### Stage 5 — Secrets store
- Per-repo secrets, AES-256-GCM at rest, decrypted only to inject into the trigger
  env. Never logged, never rendered.
- **Verify:** encrypt/decrypt round-trip; a secret never appears in logs or the
  UI; injected only into the Scaleway `start` env, not persisted in the run row.

### Stage 6 — Status UI + logs
- webui: per-repo runs list, run detail with per-job status, a log viewer
  (`render`, ANSI→HTML if needed). A commit-status affordance in the browse views
  (tasks 14/15). Admin-gated like the rest of `/ui`.
- **Verify:** route tests, golden + failure, still behind `requireAdmin`.

### Stage 7 — Self-deploy loop + latest-wins guard
- A green run on `master` redeploys gitmote (the ouroboros). **Latest-wins:** the
  deploy step no-ops unless its target SHA is still the branch tip in gitmote — a
  slow, stale run deploys nothing rather than regressing prod. The GitHub mirror
  becomes **break-glass**: an independent clean copy to redeploy from by hand if a
  bad push wedges the loop.
- **Verify:** a run whose SHA is no longer the tip deploys nothing; the leased
  writer keeps a redeploy from ever running two writers (already proven by
  `make e2e-restore` + the lease); `make e2e-local` stays green.

## Cross-cutting safety (likely a new [safety.md](../docs/architecture/safety.md) section)

- **Untrusted code execution.** Repo workflows are attacker-controlled. The
  **Scaleway Job is the isolation boundary** — the writer never executes repo
  code. The runner holds only a scoped clone token + injected secrets, reports
  over the authenticated API, and cannot reach s3lite.
- **Fire-and-forget everywhere.** CI dispatch, trigger, and deploy must never fail
  or block a push (content-before-pointer, generalized).
- **Single writer holds.** Only the leader dispatches and writes run state;
  runners funnel results back through the API. Nothing here adds a second writer.
- **Secrets** are the sole genuinely new surface — Stage 5's encrypted store, kept
  separate and never logged.

## Ops (deferred to the relevant stages; lands in [ops.md](../docs/ops.md))

- One-time `scw jobs definition create` for the runner image (cpu/memory/storage),
  mirroring pan0's worker definition; **one definition serves all repos**, env
  injected at trigger.
- New CI secrets / env: `SCW_SECRET_KEY`, `SCW_REGION`, `SCW_CI_JOB_DEFINITION_ID`,
  `WORKER_SECRET`, and the secrets-store master key — on the container, distinct
  from the deploy pipeline's GitHub Actions secrets.
